# 画面匹配方案

## 目标

用较快的粗匹配先定位原片窗口，再在局部窗口里逐帧精匹配，最终导出给剪映使用的片段时必须满足：

```text
source_frame_count == viral_frame_count
```

不要直接用 VMF 输出的小数秒作为最终剪映片段。最终片段必须用整数帧号表达。

## 输入样例

```text
爆款视频:
/Users/chaiyapeng/Downloads/对标/9/对标爆款老爸老妈的浪漫史S6E22.mp4

原片视频:
/Users/chaiyapeng/Downloads/对标/9/How.I.Met.Your.Mother.S06E22.1080p.AMZN.WEBRip.DDP5.1.x264-NOGRP.mkv
```

两边真实帧率约为：

```text
23.976 fps
```

不要用整数 `24fps` 做最终帧号换算。

## 总体流程

### 1. VMF 3fps 粗匹配

使用 `video-match-finder` 做低采样率粗定位。

仓库：

```text
https://github.com/Sinoosoida/video-match-finder
```

本机 macOS 上需要限制 OpenMP 并发，否则 `torch/faiss` 组合容易段错误：

```bash
KMP_DUPLICATE_LIB_OK=TRUE
OMP_NUM_THREADS=1
MKL_NUM_THREADS=1
VECLIB_MAXIMUM_THREADS=1
```

推荐粗匹配参数：

```bash
vmf scan \
  --fps 3 \
  --model dinov2_vits14 \
  --device cpu \
  --no-cropdetect \
  --no-mirror \
  --batch-size 64 \
  --inflight 1 \
  --legacy-ransac
```

参数说明：

```text
--fps 3
  每秒采 3 帧，只做粗定位，速度快。

--model dinov2_vits14
  DINOv2 small 模型，速度和效果比较均衡。

--device cpu
  当前 VMF 只支持 cpu/cuda。本机 torch 显示 cuda_available=False，mps_available=True，
  但 VMF 默认不支持 Apple MPS。

--no-cropdetect
  跳过黑边检测预处理，节省时间。

--no-mirror
  不建立镜像索引，减小索引量。当前样例不需要镜像粗匹配。

--batch-size 64
  每批送入模型编码的帧数。64 在本机较稳定。

--inflight 1
  同时只跑一个编码任务，避免 OpenMP 多线程冲突。

--legacy-ransac
  先用 legacy RANSAC 找粗候选。默认 smooth 在低 fps 下可能过保守。
```

本次样例耗时：

```text
VMF 3fps 索引 + 默认 legacy 搜索: 152.12s
同索引放宽搜索: 2.82s
```

### 2. 放宽粗搜索，确定局部精修窗口

3fps 默认搜索找到短候选：

```text
原片: 235s - 263s
爆款: 14s - 31s
```

基于同一个 3fps 索引，再跑放宽搜索，得到更完整粗窗口：

```text
smooth_loose:
原片: 229.0s - 309.333s
爆款: 0.333s - 73.0s
speed_ratio: 0.9125
score: 0.6639
z: 43.704
p: 0.0196
```

实际用于逐帧精修时，窗口需要向两边扩展。当前样例采用：

```text
source_window: 220s - 400s
```

这个窗口覆盖了粗匹配区间，也覆盖了后半段在原片里的候选位置。

### 3. 局部逐帧精匹配

逐帧精匹配不再用 VMF 低 fps 采样，而是按原始视频帧率读取：

```text
viral: 全片逐帧特征
source: 只读取粗窗口 source 220s - 400s 的逐帧特征
```

局部精修使用 crop-aware 特征：

```text
爆款帧:
  scale -> 108x108 -> 灰度 -> 18x18 block mean -> normalize

原片帧:
  scale -> 192x108 -> 横向裁剪 108x108
  crop_step = 40px 原始尺度附近
  同时搜索 flipped true/false
  18x18 block mean -> normalize

候选搜索:
  cKDTree top_k = 60

连续性选择:
  优先选择 source_frame 跟随 viral_frame 连续递增的候选
  offset_tolerance = 10 frames
  crop_tolerance = 400 px
```

本次逐帧精修耗时：

```text
probe: 0.22s
extract_viral_fullfps_features: 0.68s
extract_source_window_fullfps_crop_features: 16.18s
query_topk_candidates: 41.51s
temporal_choose_and_segments: 0.18s

total: 58.76s
```

`raw_segments` 里所有片段已经满足：

```text
viral_frame_count == source_frame_count
```

`filled_segments` 会用相邻片段补齐 gap，也满足帧数一致，可以用于完整覆盖爆款视频。

## 最终导出规则

剪映草稿不要用小数秒直接排片。必须先转整数帧：

```text
viral_start_frame
viral_end_frame_inclusive
source_start_frame
source_end_frame_inclusive
frame_count = viral_end_frame_inclusive - viral_start_frame + 1
```

然后强制：

```text
source_end_frame_inclusive = source_start_frame + frame_count - 1
```

导出到剪映时再把帧号换算成时间：

```text
start_seconds = frame / fps
duration_seconds = frame_count / fps
```

当前样例 fps 使用：

```text
fps = 23.976
```

如果 `ffprobe` 读到精确分数，应优先使用精确分数，例如：

```text
2997 / 125 = 23.976
```

## 注意事项

1. `24fps` 不是最精，真实帧率才是最精。
2. VMF 的低 fps 输出只能做粗定位，不能直接作为最终剪映片段。
3. 逐帧精修必须在粗窗口里做，否则全片逐帧会慢很多。
4. 当前 CPU 方案可以跑通，约 3分34秒。
5. 如果要进一步加速，有两个方向：
   - 修改 VMF 支持 Apple MPS。
   - 搭建远程 CUDA embedding endpoint，然后用 VMF `--endpoint`。
6. 生成对照视频时也要注意输出帧率，不能把 `24000/1001` 当成 `2997/125`，否则可能多 1 帧。
